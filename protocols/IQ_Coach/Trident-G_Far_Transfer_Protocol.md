# Trident-G Far Transfer Protocol

## Horizontal transfer, vertical transfer and the Gf → Gc → Gf spiral

### Mechanistic protocol specification for applied cognitive training, readiness routing and transfer-focused performance development

---

## Summary

The Trident-G Far Transfer Protocol is a structured cognitive-training and performance-routing framework designed to train **portable adaptive intelligence**, rather than narrow task improvement.

The protocol is built around four coupled processes:

```text
horizontal transfer
+ vertical transfer
+ niche deployment
+ delayed consolidation
```

The central claim is that far transfer is not produced by repeating a cognitive task until performance improves. It is produced when the learner repeatedly recovers the same relational invariant across changed task surfaces, reconstructs the same adaptive control policy across cognitive layers, deploys that policy in a meaningful real-world or simulated niche, and later re-enters the same structure after delay.

In Trident-G terms, the protocol trains the G-loop:

```text
Ψ-band access
→ entropy–mutual information regulation
→ variable abstraction
→ SR-style predictive mapping
→ global-workspace control
→ wrapper recovery
→ niche feedback
→ Gc consolidation
→ later Gf re-entry
```

The result is a **Gf → Gc → Gf spiral**:

```text
fluid cognition searches
→ useful relations are tested
→ portable structures are stabilised
→ later fluid cognition re-enters with richer material
```

This framework is designed for partner use in cognitive training, professional performance, learning support, executive decision support, human-AI reasoning workflows, education, sport, cognitive longevity and applied readiness systems.

It should not be presented as a medical, diagnostic or guaranteed IQ-gain intervention. It is best framed as an **evidence-generating transfer protocol**: a structured way to train and test whether cognitive operations survive wrapper change, interference, delay and applied use.

---

# 1. Core theoretical claim

Trident-G defines adaptive intelligence as the ability to maintain viable near-critical cognition across challenge, search relational problem spaces without collapsing into rigidity or scatter, validate useful structures across variation, and crystallise what survives into reusable Gc that can later be re-entered under new Gf demand.

The protocol is therefore built around one central training target:

```text
relational invariant abstraction
→ relational inference
→ adaptive policy
→ real-world deployment
→ delayed re-entry
```

A **relational invariant** is the structure that remains stable when surface features change. It is not the colour, symbol, arrow, word, puzzle surface or scenario itself. It is the relation that survives across those changing wrappers.

Examples include:

```text
same / different
more / less
inside / outside
before / after
cause / effect
valid / invalid constraint
state → action → outcome
path still reaches goal / path no longer reaches goal
evidence supports claim / evidence only appears to support claim
```

The key distinction is:

```text
Relational abstraction
= What is the same underneath the surface change?

Relational inference / reasoning
= What follows if that relation is true?
```

So horizontal transfer trains the learner to **recover the relation across surfaces**, while vertical transfer trains the learner to **use the relation across processing layers**.

The protocol therefore targets three kinds of transfer.

| Transfer axis           | Core question                                                                                                 | Training test                                                                                             |
| ----------------------- | ------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| **Horizontal transfer** | Does the same relational invariant survive changed surfaces?                                                  | Wrapper swaps, modality changes, near-miss foils, mixed wrappers, delayed re-checks                       |
| **Vertical transfer**   | Can the same relation be abstracted, held, inferred from, reasoned with and deployed across cognitive layers? | Readiness → control → variable abstraction → relational WMC → SR inference → mindware → action → feedback |
| **Niche transfer**      | Does the policy activate and update in the real environment?                                                  | Missions, implementation intentions, cue-action plans, real-world feedback                                |

The full transfer target is therefore:

```text
same relation
→ across wrappers
→ across layers
→ across real-world cues
→ across time
```

Or more explicitly:

```text
surface cue
→ variable abstraction
→ relational invariant abstraction
→ relational working memory
→ SR-style inference
→ explicit reasoning / mindware script
→ niche action
→ feedback
→ delayed Gc consolidation
```

The protocol is not “working-memory training” alone. Isolated working-memory training has limited evidence for reliable far transfer when separated from strategy, reasoning, transfer and application layers (Melby-Lervåg et al., 2016). In Trident-G, working memory is one layer in a wider architecture: the active workspace in which variables, relations, counterexamples, constraints, appraisals and possible future states can be held, updated and tested.

The deeper target is not simply to improve memory span. It is to train the learner to construct better relational state spaces, infer what follows from those relations, and recover those inferences under changed wrappers, interference, real-world cues and delay.

In compact form:

```text
Horizontal transfer trains relational abstraction.
Vertical transfer trains relational inference.
Niche transfer trains applied deployment.
Delayed consolidation tests whether the relation becomes reusable Gc.
```

# 2. Mechanistic background: hierarchical modular networks in a Griffiths phase

The brain is not treated here as a single homogeneous system tuned to one exact critical point. Trident-G assumes a hierarchical modular architecture in which many local modules differ in excitability, connectivity, timing, noise level and prior learning history. In such networks, critical-like dynamics can be stretched across an extended region known as a Griffiths phase, rather than existing only at a singular critical point (Moretti & Muñoz, 2013).

This matters for training. A learner does not need every part of the system to be globally critical at the same moment. Far transfer may emerge when:

```text
many local modules generate candidate fluctuations
+ global coordination selects and binds useful configurations
+ selected configurations become variables and relations
+ useful relations support inference and reasoning
+ successful structures stabilise locally
+ later fluid cognition recombines those modules
```

In the mechanistic interpretation used here:

| Signal/process                                | Trident-G interpretation                                                                              |
| --------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **Broadband arrhythmic activity (BAA)**       | Entropy-like local fluctuation field; candidate perturbations; local avalanche-like search            |
| **Local salient events**                      | Candidate micro-updates or local representational events                                              |
| **Narrowband oscillations, especially alpha** | Timing, coordination, gating and large-scale synchronisation                                          |
| **Spectral group-delay consistency (SGDC)**   | Candidate mechanism by which local events become temporally aligned into network-level salient events |
| **Fronto-parietal / global coordination**     | Mutual-information-like selection, task-relevant constraint and cross-module control                  |
| **Relational abstraction**                    | Construction of variables and relations from selected local events                                    |
| **SR-style predictive mapping**               | Organisation of relations into state → transition → consequence → next-state structure                |
| **Reasoning / mindware**                      | Explicit use of relations to infer, test, decide, reframe or act                                      |
| **Salience-autonomic regulation**             | Body-budget control over whether the Ψ-band can be sustained                                          |

Dellavale et al. (2026) provide a useful signal-level bridge. They distinguish broadband arrhythmic activity, which includes scale-free dynamics and neuronal-avalanche-like activity, from narrowband oscillations, which are linked to communication and synchronisation. Their SGDC framework suggests how local fluctuations and narrowband alignment may coalesce into transient salient events. In Trident-G terms, this supports the hypothesis that BAA-like local variability supplies candidate entropy events, while alpha/frontoparietal coordination helps bind selected events into globally usable configurations.

The Zhang–Tang model gives the computational-level analogue. Productive learning requires a balance between maximum entropy, which supports broad exploration, and mutual-information constraint, which keeps updates relevant to the task objective. Heavy-tailed update distributions then become the signature of self-organising learning rather than mere noise (Zhang & Tang, 2025).

In Trident-G, this becomes a layered information-processing sequence:

```text
local entropy events
→ candidate features
→ variable abstraction
→ relational invariant abstraction
→ SR-style predictive mapping
→ explicit inference / reasoning
→ policy selection
→ action and feedback
→ Gc consolidation
```

This sequence matters because SR-style inference cannot operate directly over raw surfaces. The learner first has to carve the state space into usable variables. Then they must identify the relation that remains stable across wrapper change. Only then can the system infer transitions, consequences and reachable future states.

The simplest distinction is:

```text
Variable abstraction
= What feature or dimension matters?

Relational invariant abstraction
= What relation stays the same when the surface changes?

SR-style inference
= What future states, consequences or action paths follow from that relation?

Explicit reasoning
= What conclusion, decision, explanation or test follows if the relation is true?
```

This gives the protocol its central training hierarchy:

```text
evidence extraction
→ variable abstraction
→ relational invariant abstraction
→ relational WMC
→ SR-style inference
→ explicit reasoning
→ niche deployment
→ delayed consolidation
```

So the Trident-G training target is:

```text
not maximum repetition
not maximum difficulty
not maximum novelty

but repeated controlled re-entry into the Ψ-band,
where entropy can reopen the search space,
mutual information can stabilise what matters,
and relational invariants can become usable predictive structures.
```

This same hierarchical modular architecture also predicts **vertical bottlenecks**: local candidate events may be generated, but fail to be transformed into variables, abstracted into invariant relations, bound in working memory, organised into predictive maps, broadcast into global-workspace control, deployed in the niche, or consolidated after delay.

In compact form:

```text
Horizontal transfer asks:
Can the same relation survive surface change?

Vertical transfer asks:
Can the relation move upward through the hierarchy
from signal to variable to relation to inference to action?

The Gf → Gc → Gf spiral asks:
Can what survives become reusable Gc
that later expands new Gf search?
```

# Part I: The Horizontal Transfer Cycle

## 3. Horizontal transfer: relational invariant recovery across wrapper change

Horizontal transfer trains the learner to recover the same **relational invariant** when the surface changes.

A wrapper is any surface format that can carry the same underlying relation or cognitive operation:

```text
arrow direction
colour relation
spatial path
symbolic rule
verbal relation
puzzle state
appraisal cue
work scenario
```

The target is not the surface itself. The target is the relation that remains stable underneath surface variation.

```text
surface A
→ relation X

surface B
→ relation X

surface C
→ relation X
```

A wrapper swap should therefore preserve the invariant while disrupting the surface policy. The purpose is not variety for its own sake. The purpose is to prevent shallow automation and force controlled recovery of the deeper relational structure.

Horizontal transfer therefore trains:

```text
surface cue
→ variable abstraction
→ relational invariant abstraction
→ wrapper change
→ invariant recovery
```

The core horizontal cycle is:

```text
Wrapper A learning
→ flattening / local asymptote
→ Wrapper B probe
→ temporary dip
→ relational invariant recovery
→ A+B mixed presentation
→ new flattening
→ Wrapper C probe
→ mixed A+B+C
→ delayed re-check
```

---

## 4. Mechanistic interpretation of the wrapper cycle

The practical wrapper cycle can be interpreted in the following mechanistic terms.

| Training event         | Mechanistic interpretation                                                                                                                                                                                                                                                                                 |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Wrapper A learning** | Local BAA/entropy-like events are increasingly constrained by task relevance. The learner becomes better at extracting the relevant variables and relation within the current surface. Where the task includes transitions or future-state structure, this also begins to tune an SR-style predictive map. |
| **Flattening**         | The local policy has become more predictable and increasingly mutual-information constrained. Performance improves less because the current wrapper-specific map is approaching its local ceiling.                                                                                                         |
| **Wrapper B probe**    | A controlled perturbation reopens local entropy search while preserving the invariant. The learner must distinguish what changed on the surface from what remained relationally stable.                                                                                                                    |
| **Temporary dip**      | The surface-specific policy is disrupted. This is useful if the deeper relation remains recoverable. A moderate dip followed by recovery is a positive transfer signal, not a failure.                                                                                                                     |
| **Recovery**           | The learner re-abstracts the invariant under the new surface. A candidate neural interpretation is that alpha/frontoparietal coordination helps rebind selected local events into a globally usable relation, allowing the active map to be re-indexed around the invariant rather than the wrapper.       |
| **A+B mixing**         | Repeated switching between wrappers prevents re-freezing into either surface-specific policy. The learner must recover the relation before the wrapper-specific response routine can dominate.                                                                                                             |
| **Foils and lures**    | Near-miss interference sharpens the boundary conditions of the invariant. Foils test whether the learner has abstracted the relation precisely rather than relying on loose similarity.                                                                                                                    |
| **Delayed re-check**   | Tests whether the structure has entered fast Gc and/or slow schematic Gc rather than remaining a session-specific or wrapper-specific trick.                                                                                                                                                               |

The ideal behavioural signature is:

```text
local improvement
→ asymptote
→ controlled perturbation
→ temporary dip
→ faster recovery
→ reduced future swap cost
→ delayed survival
```

This is not just a learning curve. It is a transfer curve.

---

## 5. Why flattening matters

Flattening means that performance in the current wrapper is improving less despite continued practice. In ordinary training, this often leads to more repetition. In Trident-G, flattening is treated as a warning sign that local fast Gc may be forming around a surface policy.

A surface policy is useful, but it is dangerous if banked too early:

```text
surface cue
→ premature rule
→ fast performance
→ poor transfer
```

The protocol therefore perturbs near local asymptote, before the surface policy becomes too stable. The aim is not to destroy learning. The aim is to test whether the learner can recover the relation when the wrapper changes.

The timing rule is:

```text
do not perturb before the invariant is minimally stable
do not wait until the surface routine is fully hardened
perturb near local asymptote
```

However, flattening should not be interpreted automatically as genuine learning saturation. It may also reflect fatigue, overload, under-engagement, poor timing quality or task confusion. For that reason, the app or programme should combine learning-curve flattening with readiness, error pattern and confidence checks before introducing a harder wrapper perturbation.

---

## 6. Why near-miss foils matter

Foils are not optional. They are the specificity engine of the protocol.

Near-miss foils prevent the learner from overgeneralising or relying on shallow familiarity. They force the learner to distinguish:

```text
same-looking but different
different-looking but same
valid relation
near relation
wrong binding
wrong lag
wrong boundary
wrong script
wrong context
```

In mechanistic terms, foils strengthen mutual-information constraint. They tell the system which distinctions matter. Without foils, entropy search may remain too broad, and wrapper recovery may reflect loose similarity rather than genuine invariant abstraction.

Foils therefore train the boundary conditions of the relation:

```text
where the relation holds
where it fails
what looks similar but is wrong
what looks different but is still the same relation
```

This is essential for transfer because a relation is only portable when its limits are understood.

---

## 7. Wrapper progression rule

The default wrapper progression is an expanding spiral:

```text
A blocked
→ A with foils
→ brief B probes
→ B blocked if recoverable
→ A/B mixed
→ A/B mixed with foils
→ brief C probes
→ C blocked if needed
→ A/B/C mixed
→ delayed mixed re-check
```

This is better than a rigid sequence because it allows the app or programme to adapt to the learner’s dynamical profile.

| Learner pattern                  | Wrapper policy                                                                           |
| -------------------------------- | ---------------------------------------------------------------------------------------- |
| **Regulated**                    | Use standard perturbation near asymptote; increase wrapper complexity gradually          |
| **Fast brittle**                 | Add slow-check prompts, disconfirming foils and post-error reset before increasing speed |
| **Overloaded**                   | Delay harder wrapper swaps; simplify, scaffold and reduce information density            |
| **Slow compensatory**            | Keep the wrapper swap but reduce time pressure and define “good enough” thresholds       |
| **Constraint-dominant / rigid**  | Perturb earlier; add boundary cases and cross-wrapper recovery                           |
| **Entropy-dominant / scattered** | Use clearer targets, stronger foils and more blocked practice before mixing              |
| **SR-depth limited**             | Add deeper relation families and temporal-horizon demands                                |
| **Consolidation-limited**        | Add delayed re-checks, rest intervals and cross-session retrieval                        |

So the app should not simply ask:

```text
Has the user improved?
```

It should ask:

```text
Has the user recovered the invariant after perturbation?
Has the recovery cost reduced over time?
Has the invariant survived foils?
Has it survived mixed-wrapper presentation?
Has it survived delay?
```

The strongest horizontal-transfer evidence is not trained-wrapper gain. It is reduced recovery cost after wrapper change, preserved lure resistance, stable mixed-wrapper performance and delayed survival.

# Part II: The Vertical Transfer Cycle

## 8. Vertical transfer: reconstructing the same adaptive policy across layers

Vertical transfer asks whether the same adaptive policy can be reconstructed from state regulation through evidence extraction, relational abstraction, predictive inference, explicit reasoning, action and feedback.

The vertical cycle is:

```text id="x6ft48"
Zone / readiness gate
→ cognitive-control bandwidth
→ variable abstraction
→ relational invariant abstraction
→ relational WMC
→ SR-style predictive mapping
→ mindware script
→ meta-epistemic / global-workspace control
→ niche deployment
→ feedback and reinforcement
→ delayed re-check
```

This is the control chain. Horizontal transfer tests whether the same relation survives changed surfaces. Vertical transfer tests whether the learner can rebuild the whole adaptive operation across layers.

The crucial point is that SR-style inference and explicit reasoning do not operate directly over raw surfaces. The learner first has to identify the relevant variable, abstract the relation that remains stable across wrapper change, hold that relation under interference, and then use it to infer what follows.

In compact form:

```text id="hquk4n"
surface signal
→ relevant variable
→ relational invariant
→ relation held in working memory
→ predicted transition / consequence
→ explicit reasoning move
→ action
→ feedback
→ consolidated policy
```

So vertical transfer is not simply a progression from “easy” to “hard” tasks. It is the progressive transformation of information:

```text id="z4l2y1"
signal
→ variable
→ relation
→ transition
→ policy
→ action
→ feedback
→ schema
```

---

## 9. Vertical layers

| Layer                                        | Protocol function                                                                   | Mechanistic role                                                                                     |
| -------------------------------------------- | ----------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| **0. Zone readiness**                        | Estimate cognitive-autonomic readiness and route training load                      | Determines whether the system can enter the Ψ-band without overload, brittleness or under-activation |
| **1. CCC / cognitive-control bandwidth**     | Train controlled evidence extraction under information-rate demand                  | Stabilises the base capacity to sample task-relevant information                                     |
| **2. Variable abstraction**                  | Identify which feature, cue or dimension matters                                    | Carves the state space into usable variables                                                         |
| **3. Relational invariant abstraction**      | Recover what stays the same across wrapper change                                   | Creates the portable target of horizontal transfer                                                   |
| **4. Relational WMC**                        | Hold, bind, update and compare variables and relations under delay and interference | Provides the active workspace for relation maintenance                                               |
| **5. SR-style predictive mapping**           | Learn state → transition → consequence → next reachable state                       | Expands the generative space of possible futures and action paths                                    |
| **6. Explicit reasoning / mindware scripts** | Provide reusable reasoning, reframing and inference tools                           | Converts lower-level relational operations into portable strategies                                  |
| **7. Meta-epistemic / GNW control**          | Select whether to open, constrain, test, reframe, slow-check, reset or act          | Broadcasts a control policy across specialised processors                                            |
| **8. Niche deployment**                      | Apply the policy in puzzle, study, work, appraisal, sport or decision contexts      | Tests whether the policy couples to real-world cues and feedback                                     |
| **9. Feedback and reinforcement**            | Use outcomes to update the policy and its deployment conditions                     | Strengthens useful cross-layer coordination and weakens brittle or misleading policies               |
| **10. Delayed consolidation**                | Re-test after time and changed wrapper                                              | Tests slow Gc extraction and later Gf re-entry                                                       |

## 10. Mechanistic interpretation of the vertical cycle

The vertical cycle can be described as a series of controlled rebindings across scale.

Each layer transforms the output of the layer below it:

```text id="ujdfby"
raw evidence
becomes selected evidence

selected evidence
becomes a variable

variables
become relational invariants

relations
become working-memory contents

working-memory contents
become predictive maps

predictive maps
become explicit reasoning policies

reasoning policies
become actions

actions
produce feedback

feedback
reinforces or revises the whole chain
```

The training target is therefore not only correctness. It is successful cross-layer coordination.

---

### 10.1 Zone readiness

The readiness gate asks:

```text id="jxshhl"
Is the learner currently in a trainable regime?
```

A learner who is overloaded may need simplification, scaffolding or recovery. A learner who is fast brittle may need slow-checking and disconfirming foils. A learner who is under-activated may need activation before high-load reasoning. A learner who is regulated can proceed into the full transfer cycle.

Mechanistically:

```text id="dyyf1w"
Zone readiness
= estimate whether the brain–body system can sustain entropy–MI competition
without falling into overload, brittle commitment or low-engagement collapse.
```

This matters because vertical transfer requires the whole hierarchy to be trainable. If the learner is outside the workable Ψ-band, later failures may reflect state regulation rather than a true reasoning or memory bottleneck.

---

### 10.2 CCC / cognitive-control bandwidth

CCC trains the base ability to extract relevant evidence under time and uncertainty.

Mechanistically:

```text id="ixtm5u"
CCC
= stabilise task-relevant sampling
+ prevent irrelevant salience capture
+ maintain useful information rate
```

This is the base layer of mutual-information control. It teaches the system to select what matters without prematurely closing the search space.

CCC does not yet train full reasoning. It trains the capacity to generate clean enough evidence for abstraction.

```text id="xqrp5l"
If evidence extraction is unstable,
variable abstraction becomes noisy.

If variable abstraction is noisy,
relational inference becomes fragile.
```

So CCC is the lower-level support condition for later relational work.

---

### 10.3 Variable abstraction

Variable abstraction asks:

```text id="q4hs9j"
Which feature, cue or dimension matters here?
```

This is the layer where the learner stops treating the task as a raw surface and begins carving the state space into usable dimensions.

Examples:

```text id="64wkbf"
absolute direction
relative direction
same / different
change / no change
valid / invalid state
action changes path / action blocks path
evidence supports claim / evidence does not support claim
```

Mechanistically:

```text id="0rel8v"
variable abstraction
= convert selected evidence into task-relevant dimensions
```

This layer is essential because SR-style inference cannot operate over undifferentiated stimuli. It needs a state space. Variable abstraction builds that state space.

Failure at this layer produces surface-bound performance:

```text id="gnew5f"
surface cue
→ premature rule
→ brittle response
```

Successful abstraction produces a usable variable:

```text id="csdwfk"
surface cue
→ relevant variable
→ candidate relation
```

---

### 10.4 Relational invariant abstraction

Relational invariant abstraction asks:

```text id="e7zuas"
What relation remains the same when the surface changes?
```

This is the hinge between horizontal and vertical transfer.

Horizontal transfer trains the learner to recover the same relation across wrappers. Vertical transfer then asks whether that relation can be held, inferred from, reasoned with and deployed.

Examples:

```text id="vmv3ty"
outwardness across radial and spiral wrappers
same-change across visual and symbolic transformations
reachability across visual path and puzzle state
constraint validity across puzzle, verbal reasoning and work decision
weak evidence across market signal, legal brief or research claim
```

Mechanistically:

```text id="s0tbq9"
relational invariant abstraction
= identify the structure that survives wrapper change
```

This is the content of transfer. Without it, the learner may become faster but not more portable.

---

### 10.5 Relational WMC

Relational WMC trains the learner to hold, update and compare variables and relations across delay.

Mechanistically:

```text id="3nynko"
relational WMC
= bind variables
+ protect relation labels from interference
+ maintain candidate structures long enough for SR inference
```

This is the active workspace layer. It does not by itself guarantee far transfer. Its role is to keep the right variables and relations available while wrapper change, foils and lures stress the system.

Examples:

```text id="5n6car"
Does the current relation match the one n-back?

Did the transformation repeat?

Is this the same relation under a different wrapper?

Is this a wrong-lag lure?

Is this a near-match but invalid relation?
```

Relational WMC is therefore not just memory span. It is controlled relational maintenance under interference.

---

### 10.6 SR-style predictive mapping

SR-style inference trains predictive maps rather than simple stimulus-response habits.

The relevant structure is:

```text id="3j3fwd"
state
→ transition
→ consequence
→ next reachable state
```

This is why SRs are central to Trident-G. Far transfer depends on learning the structure of reachable possibilities, not just the correct response in the current display. Predictive-map accounts of hippocampal function argue that hippocampal representations encode states partly in terms of likely future successor states, making them useful for flexible planning and generalisation (Stachenfeld et al., 2017).

Mechanistically:

```text id="k2pbhy"
SR training
= expand the generative map of possible next states
+ learn which transitions survive wrapper variation
+ support longer-horizon inference
```

SR-style mapping turns relational contents into predictive structure:

```text id="02s2lb"
relation
→ transition
→ expected consequence
→ reachable future
```

This is where relational abstraction becomes inference.

---

### 10.7 Explicit reasoning / mindware scripts

Mindware scripts are reusable control tools:

```text id="s0cyey"
What must be true?
What else could explain this?
What would make this wrong?
Which feature changed?
Can this path still reach the goal?
What is the smallest safe test?
```

These scripts should not be understood as verbal decorations. They are global-workspace handles. They allow a lower-level relational operation to become language-accessible, reportable, selectable and reusable.

A mindware script can implement:

| Script type                  | Entropy–MI function                  | SR / reasoning function                                       |
| ---------------------------- | ------------------------------------ | ------------------------------------------------------------- |
| **Constraint checking**      | Tighten mutual-information relevance | Identify what must hold for the path or claim to remain valid |
| **Boundary testing**         | Prevent premature closure            | Find where the current relation stops working                 |
| **Counterfactual search**    | Inject controlled entropy            | Generate alternative paths, explanations or future states     |
| **Reframing**                | Reconstruct the state space          | Change the variables over which inference operates            |
| **SR look-ahead**            | Test reachable future states         | Ask whether this path can still reach the goal                |
| **Slow-checking**            | Interrupt brittle over-commitment    | Reopen the active map after a likely policy error             |
| **Implementation intention** | Bind policy to real-world cue        | Convert a reasoning policy into cue-triggered action          |

The mindware layer therefore connects explicit reasoning with SR-style maps. It gives the learner a conscious handle for selecting and re-running useful predictive operations.

---

### 10.8 Meta-epistemic / global-workspace control

The meta-epistemic layer selects the right script.

It asks:

```text id="4k5mxd"
Do I need to open?
Do I need to constrain?
Do I need to test?
Do I need to reframe?
Do I need to slow-check?
Do I need to reset?
Do I need to act?
```

This maps naturally onto global neuronal workspace theory. GNW models propose that selected information can be amplified and made globally available to specialised processors, allowing it to guide report, memory, planning and action (Mashour et al., 2020).

In Trident-G, the global workspace is the point where an active relation or SR-style prediction becomes available as a control policy.

```text id="94wewy"
SR map
= implicit predictive structure over reachable futures

global workspace
= selected relation or path becomes globally available

mindware script
= compact language-accessible handle for controlling that structure
```

So the global-workspace layer does not replace SR inference. It selects, stabilises and broadcasts the relation or path that SR-style inference has made available.

A useful formulation is:

```text id="g1lzub"
SR maps provide the predictive landscape.
Global workspace selects and broadcasts a usable route through that landscape.
Mindware scripts provide the conscious control handles for re-entering that route.
```

This explains why prompts matter. A prompt such as “Can this path still reach the goal?” can rapidly re-index the active SR map by changing which states, transitions and constraints are currently sampled. But durable learning still requires outcome feedback, wrapper variation and delayed consolidation.

---

### 10.9 Niche deployment

A policy has not fully transferred until it is usable in the environment where it matters.

Niche deployment tests whether the learner can recognise the relevant cue, apply the policy under real constraints, act, and learn from feedback.

```text id="e170ox"
real-world cue
→ policy activation
→ action
→ feedback
→ update
→ delayed reuse
```

Implementation intentions support this layer because they bind a situational cue to a planned response. Meta-analytic evidence suggests that implementation intentions reliably improve goal attainment across domains (Gollwitzer & Sheeran, 2006). In Trident-G terms, they increase niche coupling by making the policy more likely to fire in the environment where it matters.

Niche deployment is where the protocol stops being only cognitive training and becomes adaptive performance training. The policy must work under real cues, noise, time pressure, incentives, social constraints and feedback.

---

### 10.10 Feedback and reinforcement across levels

Feedback is not only a final score. It is the mechanism by which the system learns which parts of the vertical chain were useful.

In reinforcement-learning terms, outcome feedback can generate prediction-error signals that update expectations about future states, action values and policy usefulness. Dopamine prediction-error models provide a plausible biological account of how unexpected outcomes can reinforce or revise policies (Schultz et al., 1997).

In Trident-G, the reinforced unit is not just the final response. The reinforced unit is the whole cross-layer sequence:

```text id="a7o5wo"
state readiness
→ evidence sampling
→ variable abstraction
→ relational invariant abstraction
→ relational WMC
→ SR-style inference
→ mindware selection
→ action
→ outcome
```

This is vertical credit assignment.

The app or programme should therefore reinforce:

```text id="rwfnq6"
accurate evidence sampling
correct variable selection
valid relation abstraction
lure resistance
successful transition prediction
appropriate mindware selection
well-timed slow-checking
useful action
informative feedback use
delayed recovery
```

Different feedback signals should update different layers.

| Feedback event                           | Layer updated                               |
| ---------------------------------------- | ------------------------------------------- |
| Error on close majority                  | CCC / evidence extraction                   |
| Confusion between surface and relation   | Variable or relational abstraction          |
| Wrong-lag lure                           | Relational WMC                              |
| Incorrect path prediction                | SR-style map                                |
| Correct answer with wrong explanation    | Global-workspace / explicit reasoning layer |
| Wrong script choice                      | Meta-epistemic control                      |
| Failure to act in real context           | Niche deployment                            |
| Same-session success but delayed failure | Consolidation / slow Gc                     |

This prevents the protocol from becoming a shallow reward loop based only on speed or correctness. The target is to reinforce the adaptive control chain that produced the answer.

---

### 10.11 Delayed consolidation

Delayed re-checks test whether the relation or policy has become portable Gc.

There are two kinds of Gc:

| Gc type               | Mechanism                                                        | Transfer expectation                |
| --------------------- | ---------------------------------------------------------------- | ----------------------------------- |
| **Fast Gc**           | Efficient compiled strategy with boundary conditions             | Useful but may remain narrow        |
| **Slow schematic Gc** | Cross-context invariant installed into representational geometry | Stronger candidate for far transfer |

The slow pathway requires time, rest and repeated cross-context recovery. Without delay, the app may be measuring temporary priming rather than genuine transfer.

Delayed consolidation asks:

```text id="eaq43u"
Can the relation be recovered later?
Can it survive a changed wrapper?
Can it survive foils after delay?
Can the right mindware script be selected without prompting?
Can the policy re-enter real-world use?
```

Only then should the structure be treated as a slow Gc candidate.

The full vertical transfer cycle can be compressed as:

```text id="4ox6mv"

CCC stabilises evidence extraction.

```

Or, in one line:

```text id="4mk6t7"
signal → variable → relation → prediction → policy → action → feedback → schema
```

The central vertical-transfer question is therefore:

```text id="maq4j0"
Can the learner reconstruct the same useful relation
from state readiness through inference, reasoning, action, feedback and delayed reuse?
```

# Part III: Vertical Bottlenecks in Hierarchical Modular Processing

## 11. Why bottlenecks matter

Vertical transfer depends on the successful movement of information through a hierarchical modular system.

At the lowest levels, local cortical modules generate candidate activity patterns. In a Griffiths-phase interpretation, these modules are not all tuned to one exact critical point. Instead, because brain networks are hierarchically modular and structurally heterogeneous, critical-like dynamics can extend across a wider region, allowing different local modules to sit near criticality at different times (Moretti & Muñoz, 2013).

In Trident-G terms, this means the brain can support many local pockets of candidate search:

```text
local module
→ local fluctuation / avalanche-like event
→ candidate variable, feature, relation or transition
```

Neuronal avalanche work is relevant here because avalanches describe cascades of neural activity whose size distributions can show scale-free organisation (Beggs & Plenz, 2003). The Trident-G interpretation does not require every training effect to be a literal avalanche. The safer claim is that local avalanche-like or BAA-like dynamics are plausible carriers of local entropy search.

However, local candidate events are not enough. Far transfer requires useful local events to be selected, bound and made available to wider control systems.

```text
local event
→ temporal alignment
→ cross-module binding
→ frontoparietal / global-workspace access
→ controlled action or strategy
```

A vertical bottleneck occurs when information is available at one level but cannot be transformed, bound, integrated or deployed at the next.

## 12. Bottlenecks by vertical layer

| Bottleneck layer                      | Mechanistic interpretation                                                                                                                                           | Likely app or programme signature                                                               | Training response                                                                      |
| ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **0. State / Zone readiness**         | The system cannot reliably enter or sustain the Ψ-band. Local search and global control are compromised by overload, brittleness, low vigilance or autonomic strain. | Low engagement-vigilance, unstable RTs, high personal deviation, poor recovery.                 | Activate, recover, simplify, reduce load or slow-check before deeper training.         |
| **1. CCC / evidence extraction**      | Local sensory/evidence events are not being sampled with enough precision or speed. The bottleneck is before relational abstraction.                                 | Low Cognitive Bandwidth, high timing cost, errors on close majorities.                          | Train controlled evidence extraction with easier wrappers and calibrated exposure.     |
| **2. Variable abstraction**           | The learner has not carved the state space into usable variables. SR inference has weak material to operate on.                                                      | Good simple extraction but poor Frame Bandwidth or high Frame Cost.                             | Use foils, contrastive examples and “Which feature changed?” prompts.                  |
| **3. Relational WMC**                 | Variables can be identified, but cannot be held, updated or compared under delay/interference.                                                                       | Poor Frame Memory, high wrong-lag lures, fragile n-back level.                                  | Train relation memory, lure resistance and slower mixed-wrapper relation maintenance.  |
| **4. SR / path prediction**           | Relations are held locally, but do not become predictive maps over future states or consequences.                                                                    | Poor Change Tracking or Path Prediction despite adequate WMC.                                   | Train state → transition → consequence → reachable-state tasks.                        |
| **5. Global integration / workspace** | Local relations exist but do not become globally available as a controllable strategy.                                                                               | Performs in implicit/puzzle format but fails verbal explanation, mixed reasoning or prompt use. | Use mindware scripts, explanation prompts and cross-format recovery.                   |
| **6. Meta-epistemic selection**       | The learner has tools but chooses the wrong mode: opens when they should constrain, or constrains when they should reframe.                                          | Wrong-script lures, brittle over-checking, scattered exploration or premature closure.          | Train mode selection: open, constrain, test, reframe, reset or act.                    |
| **7. Niche deployment**               | The policy works in-app but does not couple to real-world cues and feedback.                                                                                         | Good task scores but poor mission/application transfer.                                         | Use implementation intentions, real-world cue mapping and feedback logs.               |
| **8. Delayed consolidation**          | Fast performance is present, but slow schematic Gc is not installed.                                                                                                 | Good same-session transfer, poor next-day or changed-wrapper recovery.                          | Add delayed probes, rest spacing, cross-session retrieval and sleep/recovery emphasis. |

## 13. Mechanistic summary of vertical bottlenecks

The bottleneck model can be summarised as:

```text
local avalanches / BAA-like fluctuations
= candidate entropy events

hierarchical modular structure
= many semi-independent local search spaces

alpha/frontoparietal synchrony
= candidate temporal integration and control-binding mechanism

global workspace
= selected relation becomes available for reasoning, report and action

SR map
= relation becomes predictive over future states

Gc consolidation
= repeated successful structures become reusable modules

later Gf
= re-enters the Ψ-band with richer modular material
```

Vertical transfer is therefore not simply “more difficult tasks stacked upward”. It is cross-scale information transformation:

```text
signal
→ variable
→ relation
→ transition
→ policy
→ action
→ feedback
→ schema
```

A vertical bottleneck is where that transformation fails.

## 14. Practical bottleneck detection rule

The app or training system should infer bottlenecks relatively, not absolutely. A bottleneck is not a diagnosis. It is a layer that appears to constrain later layers relative to the person’s own baseline and the available norm group.

A simple rule is:

```text
If layer k is weak,
and upstream layers are adequate,
and downstream layers repeatedly fail,
then flag layer k as a potential bottleneck.
```

Examples:

```text
Good Cognitive Bandwidth
+ poor Frame Bandwidth
= possible variable-abstraction / relational extraction bottleneck.

Good Frame Bandwidth
+ poor Frame Memory
= possible relational WMC bottleneck.

Good Frame Memory
+ poor Path Prediction
= possible SR-horizon bottleneck.

Good Path Prediction
+ poor Reasoning Transfer
= possible global-workspace / explicit-policy bottleneck.

Good same-session transfer
+ poor delayed recovery
= possible consolidation bottleneck.
```

Partner-facing user language should remain cautious:

```text
Potential bottleneck: Frame Memory.
Your relation extraction looks stable, but relations are not yet surviving delay and lure pressure.
Suggested next block: relational memory with lower speed pressure and higher lure-control feedback.
```

The conceptual bridge is:

```text
Horizontal transfer tests whether an invariant survives surface change.
Vertical bottleneck analysis asks where the invariant fails to move upward through the hierarchy.
```


# Part V: The Gf → Gc → Gf Spiral

## 17. Why transfer is a spiral, not a loop

The combined horizontal and vertical cycles produce a **Gf → Gc → Gf spiral**.

A normal practice loop looks like this:

```text
task practice
→ performance improves
→ routine becomes faster
→ more task practice
```

That is flat optimisation. The learner becomes more fluent inside the trained surface, but the underlying cognitive architecture may not become more portable.

The Trident-G spiral is different:

```text
Gf search
→ variable abstraction
→ relational invariant discovery
→ SR-style inference
→ wrapper validation
→ vertical policy reconstruction
→ niche feedback
→ reinforcement across levels
→ fast and slow Gc banking
→ later Gf re-entry from a richer starting point
```

The learner does not simply return to the same task with better fluency. They return to the next problem with a richer representational landscape: better variables, stronger relational invariants, deeper predictive maps, clearer boundary conditions, more reusable mindware scripts and better cue-linked deployment policies.

The spiral is therefore not just:

```text
practice
→ memory
→ more practice
```

It is:

```text
search
→ abstraction
→ inference
→ validation
→ consolidation
→ richer future search
```

This is the central far-transfer claim. Transfer improves when the learner does not merely automate a response, but converts a useful relation into a modular predictive-control structure that can later be recombined under new Gf demand.

---

## 18. Gf phase: near-critical relational search

During Gf, the learner searches a near-critical relational space.

Mechanistically:

```text
local BAA / entropy-like events
→ candidate features and variables
→ alpha/frontoparietal coordination
→ selected configurations become globally available
→ relational invariant abstraction
→ mutual-information constraint tests relevance
→ SR-style inference tests what follows
```

The system should be flexible enough to generate alternatives, but constrained enough to test them. This is the Ψ-band.

In this phase, Gf does three jobs:

```text
1. Open the search space.
2. Find candidate variables and relations.
3. Test whether any relation explains, predicts or transfers.
```

The key intermediate step is **relational invariant abstraction**:

```text
surface cue
→ relevant variable
→ relation that survives surface change
```

Without this step, Gf may produce variety but not transfer. The learner may explore many alternatives without finding the structure that can be reused.

The next step is **SR-style inference**:

```text
relation
→ transition
→ consequence
→ reachable future state
```

This is where the abstracted relation becomes useful. It is no longer only “what is the same?” It becomes “what follows if this relation holds?”

So the Gf phase can be summarised as:

```text
entropy opens the search space
MI constraint selects what matters
relational abstraction identifies the invariant
SR-style inference tests what follows
global workspace makes the selected structure usable
```

Failure modes include:

| Failure                         | Description                                                                                   |
| ------------------------------- | --------------------------------------------------------------------------------------------- |
| **Too much constraint**         | Premature closure, brittle Gc, surface-specific automation                                    |
| **Too much entropy**            | Scattered exploration, poor convergence, no stable structure                                  |
| **Poor variable abstraction**   | The learner cannot identify which dimension matters                                           |
| **Poor relational abstraction** | The learner sees features but not the invariant relation                                      |
| **Poor SR inference**           | The relation is noticed but not used to predict transitions, consequences or reachable states |
| **Too much load**               | Overload, lapses, noisy control                                                               |
| **Too much speed/commitment**   | Fast brittle errors and poor reset                                                            |
| **Too little activation**       | Under-engagement and poor access to the training regime                                       |

---

## 19. Gc phase: modular stabilisation

When a relation survives foils, wrapper swaps, mixed presentation, feedback and delay, it can be stabilised as Gc.

But the target is not narrow Gc. The target is **modular, bounded, portable Gc**.

A useful Gc module should include:

```text
a relational invariant
+ the variables it operates over
+ its boundary conditions
+ its foils and near-misses
+ its valid wrappers
+ its SR-style transition structure
+ its mindware or reasoning handle
+ its deployment cues
+ its feedback history
+ its delayed recovery signature
```

So Gc is not merely stored knowledge. In the Trident-G protocol, Gc is a compact predictive-control module.

Examples:

```text
“outwardness”
+ valid radial / spiral wrappers
+ wrong-wrapper lures
+ relation-memory stability
+ delayed recovery

“path still reaches the goal”
+ transition map
+ invalid near-path foils
+ SR look-ahead prompt
+ next-move policy

“weak evidence relation”
+ boundary between support and appearance of support
+ disconfirmation prompt
+ professional decision cue
+ delayed reuse under changed context
```

In hierarchical modular networks, local modules can stabilise useful structures without forcing the whole system into rigid global closure. This is why Griffiths-phase organisation is important. It allows local pockets of stabilisation while preserving wider flexibility.

So Gc is not the enemy of Gf. Good Gc gives Gf richer material.

The key distinction is:

```text
bad Gc
= surface routine that closes the search space too early

good Gc
= bounded predictive module that can be recombined later
```

In Trident-G terms, good Gc should reduce unnecessary search while preserving future re-entry into the Ψ-band.

---

## 20. Reinforcement and Gc banking

Gc banking should depend on more than same-session success.

A relation or policy should only be treated as a strong Gc candidate when the system has received converging evidence across levels:

```text
accurate evidence extraction
+ correct variable abstraction
+ invariant recovery across wrappers
+ lure resistance
+ successful SR-style prediction
+ appropriate mindware selection
+ useful action
+ informative feedback
+ delayed recovery
```

This is reinforcement across the vertical hierarchy.

The reinforced unit is not just the final answer. It is the whole adaptive chain:

```text
state readiness
→ evidence sampling
→ variable abstraction
→ relational invariant abstraction
→ relational WMC
→ SR-style inference
→ mindware selection
→ action
→ feedback
→ delayed reuse
```

This matters because the same correct answer can be produced by different mechanisms.

| Same outward result                             | Different underlying mechanism |
| ----------------------------------------------- | ------------------------------ |
| Correct answer by surface shortcut              | Low transfer value             |
| Correct answer by explicit relation recovery    | Higher transfer value          |
| Correct answer after wrapper change and foils   | Stronger transfer value        |
| Correct answer after delay and niche deployment | Strong Gc candidate            |

The protocol should therefore reinforce **how** the learner arrived at the answer, not just whether the answer was correct.

In compact terms:

```text
reward the recoverable control chain,
not just the response.
```

---

## 21. Gf re-entry: recombination from a richer map

Later Gf re-enters the Ψ-band with more structured material available.

A successful prior cycle leaves behind:

```text
better variable abstractions
clearer relational invariants
richer SR maps
stronger lure resistance
more portable mindware scripts
better global-workspace handles
better implementation cues
clearer boundary conditions
more reliable delayed recovery
```

This expands the generative space for the next problem. Fluid cognition now searches over better building blocks.

The role of global workspace is important here. The SR map provides the predictive landscape, but the global workspace makes selected relations and routes available for reasoning, report, planning and action.

So the re-entry sequence is:

```text
stored Gc module
→ reactivated by new cue
→ selected into global workspace
→ used to re-index the active SR map
→ recombined with new variables and relations
→ new Gf search begins from a richer state
```

This is why the spiral matters. Prior Gc does not merely make old actions faster. It changes what later Gf can search.

The spiral is therefore:

```text
Gf opens the search space.
Horizontal transfer isolates the invariant.
Vertical transfer binds the control chain.
Niche transfer couples the policy to real feedback.
Reinforcement selects the useful cross-layer sequence.
Gc stabilises what survives.
Later Gf recombines the enriched modules.
```

In compact form:

```text
Gf → tested relation → predictive Gc module → richer Gf
```

Or, more fully:

```text
near-critical search
→ relational abstraction
→ predictive inference
→ wrapper validation
→ global-workspace control
→ niche feedback
→ Gc banking
→ later near-critical re-entry
```

---

## 22. Practical implication for the protocol

The Gf → Gc → Gf spiral gives the app or programme a simple rule:

```text
Do not bank fluency.
Bank portable structure.
```

A structure should not be treated as consolidated merely because performance improved. It should be banked only when it survives:

```text
wrapper change
near-miss foils
mixed presentation
SR-style prediction
explicit reasoning or mindware selection
niche deployment where relevant
delayed re-check
```

The practical scoring implication is:

```text
trained-task gain
= local improvement

wrapper recovery
= horizontal transfer evidence

successful prediction / reasoning
= vertical transfer evidence

cue-linked real-world use
= niche transfer evidence

delayed recovery
= Gc consolidation evidence
```

The strongest evidence for the spiral is a reduction in future search cost without loss of flexibility:

```text
the learner needs less search
but can still reopen search when the context changes
```

That is the target state of the Trident-G Far Transfer Protocol.

In one sentence:

**The Gf → Gc → Gf spiral trains adaptive intelligence by using near-critical search to discover relational invariants, validating those invariants across wrappers and cognitive layers, reinforcing the successful control chain, stabilising what survives as bounded predictive Gc, and later re-entering Gf with richer modular material for new inference.**

# Part VI: Practical Protocol Rules

## 23. Rule 1: Do not reward trained-task gain alone

A score should not be treated as evidence of transfer merely because it improved inside one wrapper.

The app or programme should distinguish:

| Improvement type                  | Meaning                                                             |
| --------------------------------- | ------------------------------------------------------------------- |
| **Trained-wrapper gain**          | Better at the local task                                            |
| **Variable abstraction**          | Better at identifying which feature or dimension matters            |
| **Relational invariant recovery** | Same relation recovered after surface change                        |
| **Mixed-wrapper stability**       | Relation survives unpredictable wrapper switching                   |
| **Lure resistance**               | Learner resists tempting near-matches                               |
| **SR-style prediction**           | Relation supports transition, consequence or reachability inference |
| **Mindware selection**            | Learner chooses the right reasoning or reframing tool               |
| **Niche deployment**              | Policy works in a real or simulated environment                     |
| **Delayed recovery**              | Structure survives time and changed context                         |

The transfer score should privilege the latter layers over trained-wrapper gain. Local improvement matters, but it should not be treated as transfer unless it survives wrapper change, foils, inference demand, policy selection and delay.

---

## 24. Rule 2: Perturb near local asymptote

The app should introduce wrapper change when:

```text
recent slope is near zero
accuracy is stable in the training band
minimum trials are complete
lure performance is acceptable
confidence is sufficient
readiness state is suitable
```

But the app should not wait for prolonged overtraining.

The aim is to perturb the surface when the relation is minimally stable, but before the learner has banked a narrow wrapper-specific routine.

Do not perturb simply because the user has completed enough trials. Perturb when the pattern suggests that a useful local relation has stabilised and is ready to be tested under changed surface conditions.

---

## 25. Rule 3: Use perturbation probes before full switches

A wrapper B probe is a small transfer test, not a full replacement.

Example:

```text
A A A B A A B A
```

If B causes a moderate dip and recovery, continue. If B causes collapse, return to A with scaffolding and explicit invariant prompts.

Useful prompts include:

```text
Which feature changed?
What relation stayed the same?
What would make this a near-miss rather than a true match?
Can the same relation still guide the next step?
```

This turns the wrapper probe into both a transfer test and a relational abstraction exercise.

---

## 26. Rule 4: Mix wrappers progressively

Use progressive mixing:

```text
80/20
→ 60/40
→ 50/50
→ random mixed
→ mixed with foils
→ delayed mixed
```

Random mixing is the strongest test because the learner cannot rely on wrapper-specific preparation. They must recover the relation before the surface-specific response routine can dominate.

Mixed-wrapper stability should therefore be scored separately from blocked-wrapper improvement.

---

## 27. Rule 5: Treat foils as part of the target

Foils should be introduced early enough to prevent broad, fuzzy learning.

Examples:

```text
wrong lag
wrong binding
same surface, wrong relation
same relation, wrong wrapper
near boundary
wrong transition
wrong consequence
wrong script
wrong context
misleading cue
```

Foils do not merely make the task harder. They define the boundary conditions of the relation.

The learner should know:

```text
where the relation holds
where it fails
what looks similar but is wrong
what looks different but is still the same relation
```

This is essential for transfer because a relation is only portable when its limits are understood.

---

## 28. Rule 6: Route by state

The same perturbation is not suitable for every state.

| Readiness pattern     | Training route                                                        |
| --------------------- | --------------------------------------------------------------------- |
| **Regulated**         | Full horizontal and vertical cycle                                    |
| **Slow compensatory** | Keep structure, reduce unnecessary checking, gradually increase speed |
| **Overloaded**        | Simplify, scaffold, reduce entropy and load                           |
| **Fast brittle**      | Slow-check, disconfirming foils, post-error reset                     |
| **Low vigilance**     | Activate, recover, or use easier entry task                           |
| **Autonomic strain**  | Shorten block, reduce intensity, schedule recovery                    |

State routing protects transfer quality. A failed wrapper probe may mean the relation is not portable, but it may also mean the learner was overloaded, under-activated, brittle or physiologically strained. The app should therefore interpret transfer failures in relation to readiness state.

---

## 29. Rule 7: Route by bottleneck layer

Do not only ask whether the user failed. Ask where the vertical chain failed.

```text
signal
→ variable
→ relation
→ transition
→ policy
→ action
→ feedback
→ schema
```

If the likely bottleneck is early in the chain, simplify or train prerequisites. If the likely bottleneck is late in the chain, train policy selection, niche deployment or delayed recovery.

Examples:

```text
poor evidence extraction
→ train CCC

good evidence extraction but poor relation recovery
→ train variable and relational abstraction

good relation recovery but poor wrong-lag performance
→ train relational WMC

good relation memory but poor reachability prediction
→ train SR-style inference

good inference but poor explanation or strategy use
→ train mindware and global-workspace control

good in-app performance but poor applied use
→ train niche deployment

good same-session performance but poor delayed recovery
→ train consolidation and retrieval
```

This prevents the app from treating all failure as one generic weakness.

---

## 30. Rule 8: Reinforce the route, not just the response

Correct answers can arise from different mechanisms.

| Same outward result                              | Different underlying route     |
| ------------------------------------------------ | ------------------------------ |
| Correct by surface shortcut                      | Low transfer value             |
| Correct by relation recovery                     | Higher transfer value          |
| Correct after wrapper change and foils           | Stronger transfer value        |
| Correct after prediction, reasoning and feedback | Strong vertical-transfer value |
| Correct after delay and niche deployment         | Strong Gc candidate            |

The app should therefore reinforce the cross-layer sequence that produced the answer:

```text
readiness
→ evidence sampling
→ variable abstraction
→ relational invariant abstraction
→ relational WMC
→ SR-style inference
→ mindware selection
→ action
→ feedback
→ delayed reuse
```

This is the practical reinforcement-learning rule. Do not reward only speed and correctness. Reward the recoverable control chain.

---

## 31. Rule 9: Bank only what survives delay

A relation or policy should not count as slow schematic Gc unless it survives:

```text
changed wrapper
near-miss foils
mixed presentation
SR-style prediction
explicit reasoning or mindware selection
delay
and, where relevant, niche deployment
```

This is the anti-practice-effect rule.

The strongest candidate for slow schematic Gc is not a fluent routine. It is a bounded predictive-control module:

```text
relation
+ variables
+ boundary conditions
+ foils
+ valid wrappers
+ transition structure
+ reasoning handle
+ deployment cues
+ feedback history
+ delayed recovery
```

# Part VII: Example Session Logic

## 32. Example: relational WMC wrapper cycle

A relational n-back target might begin with radial in/out relations.

```text
Wrapper A: radial in/out
Foils: wrong-lag lure, same wrapper/wrong relation
Flattening: n-back level stabilises, slope falls
Wrapper B probe: tangential clockwise/anticlockwise
Dip: accuracy falls temporarily
Recovery: relation memory stabilises in B
A+B mix: random radial/tangential trials
Foils: same relation/wrong wrapper, response-mapping lure
Delayed probe: next-session mixed radial/tangential test
Wrapper C: spiral in/out
A+B+C mix: full polar-wrapper recovery
```

Mechanistically:

```text
variable abstraction
→ relation binding
→ perturbation
→ active SR re-indexing where relevant
→ invariant recovery
→ portable relational memory
```

This example mainly trains the active relational workspace. It tests whether the user can extract a relation, hold it across delay, resist lures and recover it when the wrapper changes.

---

## 33. Example: SR-style path prediction cycle

A path-prediction target might begin with a simple visual path or puzzle state.

```text
Wrapper A: visual path
Target relation: current state still reaches goal
Foils: near path that looks promising but blocks later
Flattening: reachability judgement stabilises
Wrapper B probe: symbolic graph
Dip: accuracy falls temporarily
Recovery: reachability relation returns in symbolic form
A+B mix: visual and symbolic reachability trials
Wrapper C: puzzle state or work-flow decision
Delayed probe: next-session changed-wrapper reachability test
```

Mechanistically:

```text
relation
→ transition
→ consequence
→ reachable future state
→ SR-style prediction
→ policy choice
```

This example trains the bridge from relational abstraction to predictive inference.

---

## 34. Example: mindware wrapper cycle

A mindware target might be boundary testing.

```text
Script: “What would make this wrong?”
Wrapper A: abstract puzzle
Wrapper B: verbal reasoning problem
Wrapper C: work decision
Wrapper D: emotional appraisal
Foils: wrong use of boundary testing when action is needed
Delayed probe: apply the same script to a new real-world cue
```

Mechanistically:

```text
local strategy use
→ global-workspace control handle
→ SR-map re-indexing where relevant
→ cross-context policy recovery
→ implementation cue
→ niche feedback
```

This example trains the explicit control layer. The learner is not just solving a task. They are learning when to select a reasoning policy and how to redeploy it under changed conditions.

---

## 35. Example: professional decision-readiness use

A professional user may not need a long training session before every work block. The protocol can also support state-contingent decision routing.

Example:

```text
brief readiness gate
→ cognitive-control profile
→ bottleneck estimate
→ recommended work mode
```

Possible routes:

| Readout                                                     | Work recommendation                                                 |
| ----------------------------------------------------------- | ------------------------------------------------------------------- |

| **Autonomic strain with good cognition**                    | Shorten block and schedule recovery                                 |
| **Good extraction but weak reasoning transfer**             | Use explicit mindware prompts and decision templates                |
| **Good same-session performance but weak delayed recovery** | Schedule delayed re-checks and reduce novelty stacking              |

This allows professional partners to use the protocol as a decision-readiness and training-route layer, not merely as a cognitive game.

# Part VIII: Claims Boundary and Validation Plan

## 32. Claims boundary

The Trident-G Far Transfer Protocol is an evidence-generating protocol, not a fully validated commercial proof of IQ gain.

The defensible claim is:

> The protocol is designed to train and test the portability of relational control policies across wrappers, cognitive layers, real-world cues and delay.

The stronger claim still requiring direct validation is:

> The complete Trident-G stack produces reliable far transfer to broad intelligence outcomes beyond active-control comparison conditions.

The evidence base supports the component rationale: critical-like and Griffiths-phase dynamics are plausible for hierarchical brain networks; SRs provide a strong model of predictive relational maps; global workspace theory supports broadcast-style control; implementation intentions support cue-linked deployment; inductive reasoning training has shown promising transfer effects; isolated WM training has weaker far-transfer evidence. The integrated Trident-G protocol should therefore be treated as a coherent mechanistic hypothesis requiring staged validation.

## 33. Partner validation targets

Professional or industry partners can evaluate the protocol at several levels.

```text
near transfer
wrapper transfer
mixed-wrapper recovery
delayed recovery
bottleneck stability
reasoning transfer
niche deployment
longitudinal baseline shift
comparison against active controls
```

The most important validation question is not:

```text
Did the user improve at the trained game?
```

It is:

```text
Did the trained relation or policy survive wrapper change,
foil interference,
vertical reconstruction,
applied use,
and delayed re-entry?
```

## 34. Practical partner implementation model

A partner implementation can begin with a staged approach. This allows the protocol to be adopted gradually, while measuring which transfer layers are working before stronger claims are made.

| Stage                                             | Purpose                                                                                                                        |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **Stage 1: Readiness and baseline**               | Establish personal baseline, state-routing validity, cognitive-control profile and readiness patterns                          |
| **Stage 2: Component foundations**                | Train CCC, variable abstraction, relational invariant abstraction, relational WMC and SR-style inference separately            |
| **Stage 3: Horizontal wrapper transfer**          | Test whether the same relation can be recovered across changed task surfaces, foils, lures and mixed wrappers                  |
| **Stage 4: Vertical integration and reasoning**   | Add mindware scripts, meta-epistemic control, global-workspace prompts, reasoning bridges and feedback-based credit assignment |
| **Stage 5: Niche deployment**                     | Test whether the policy activates in real-world or simulated professional contexts through cues, action and feedback           |
| **Stage 6: Delayed consolidation and Gc banking** | Test next-day or later recovery, changed-wrapper re-entry, delayed lure resistance and update future training routes           |

This makes the protocol commercially usable without overclaiming. Partners can adopt the full architecture gradually, while measuring whether transfer is occurring at the level of task performance, wrapper recovery, relational abstraction, predictive inference, reasoning policy, applied deployment or delayed consolidation.
