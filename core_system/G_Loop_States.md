## State framework

The best version is now a **three-channel loop-readiness model**:

```text
Cognitive-control state
+ engagement-vigilance state
+ autonomic/body-budget state
+ personal-baseline deviation
= predicted G-loop vulnerability
```

So Zone Coach should not simply say:

```text
You are regulated / overloaded / brittle / slow.
```

It should say:

```text
Given your current mind, vigilance and body-budget pattern,
which part of the G-loop is most likely to fail next?
```

## 1. Cognitive-control states as G-loop control styles

| Zone Coach cognitive state   | G-loop meaning                                                                                                                   | Most vulnerable G-loop phases                                   | Main risk                                                                             | Corrective route                                                                         |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------- | ------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| **Regulated / in-zone-like** | The system can run the full adaptive loop: estimate state, protect F★, model the situation, choose scale, close, act, evaluate and bank. | Low vulnerability unless task is poorly chosen.                 | Wasting a good state on trivial closure or shallow banking.                           | Use the state for meaningful Gf modelling, right-sized closure and deep Gc banking.      |
| **Slow compensatory**        | Control is preserved but costly. The system can model and check, but action and closure become slow.                                     | Scale, closure, action, banking.                                | Over-checking, delayed closure, excessive caution, shallow “safe” outputs.            | Define “good enough”; time-box; choose reversible closure; bank a simple portable rule.  |
| **Globally overloaded**      | Current demand exceeds stabilising capacity. The loop is too large for the available state/support.                                      | State estimate, objective, Gf inference, scale, closure.        | Wrong variables, poor model, fragmented action, open-loop accumulation, F★ erosion.   | Reduce scale; simplify; scaffold; recover; avoid high-stakes irreversible action.        |
| **Fast brittle**             | Action policy outruns modelling and checking. The system moves quickly but does not sufficiently test the model.                         | Objective, Gf inference, pre-mortem, feedback, deep Gc banking. | Premature closure, weak disconfirmation, error-bursty action, banking the wrong rule. | Slow-check; ask “what would make this wrong?”; use checklists; delay irreversible moves. |

The major update is that **fast brittle** is now especially a **Gf-to-action failure**: it can appear efficient, but closure is not sufficiently inference-grounded. The G-loop explicitly states that closure should follow from the Gf situation model, not impulse or surface habit. 

## 2. Vigilance as a loop-maintenance gate

Vigilance should not be treated as a fifth cognitive-control class. It is a **gate on whether the loop can be sustained**.

| Vigilance state              | G-loop meaning                                                                            | Risk                                                                  | Routing implication                                                                                      |
| ---------------------------- | ----------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **Stable vigilance**         | The system can probably sustain attention long enough for modelling, action and feedback. | Depends mainly on cognitive-control and autonomic state.              | Allow normal loop scale if cognitive and body-budget channels also support it.                           |
| **Low / unstable vigilance** | The loop may not remain coherent across time.                                             | Lapses, missed constraints, weak feedback registration, poor banking. | Reduce loop scale; use activation or recovery; avoid long-horizon Gf modelling until alertness improves. |

So low vigilance mainly impairs:

```text
State → Gf inference → SR horizon → Action consistency → Feedback → Deep Gc
```

It particularly weakens **SR horizon selection** because the person may not be able to hold a path model across the required time span. It also weakens deep Gc banking because the episode may not be encoded as a clear reusable rule.

## 3. Autonomic state as F★ / support / body-budget constraint

The autonomic layer should map primarily onto:

```text
Step 0: State and support estimate
Step 1: Protect F★
Step 5: Select scale and SR horizon
Step 8: Act externally
Step 9: Measure recovery / position change
```

It should **not** be mapped directly onto the cognitive taxa. The existing Zone Coach rationale already treats RR/HRV as a body-budget layer: adaptive reserve, allostatic regulation and challenge response, not as a direct cognitive state label. 

| Autonomic pattern                     | G-loop meaning                               | Risk                                                                   | Routing implication                                                     |
| ------------------------------------- | ---------------------------------------------------- | ---------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Good adaptive reserve**             | There is physiological room to run a demanding loop. | Low, if cognition and vigilance are also stable.                       | Proceed or increase challenge.                                          |
| **Low reserve / elevated activation** | The loop may complete, but at cost to F★.            | Compensatory strain, recovery debt, later collapse.                    | Shorten the loop; protect recovery; avoid stacking too many open loops. |
| **Settling / recovery trajectory**    | The body-budget is returning towards baseline.       | May still be unsuitable for immediate escalation if cognition is weak. | Use lower-load closure, recovery, or a gentle return to work.           |
| **Rising mobilisation / escalation**  | The system is still ramping up.                      | Brittle action, urgency, poor pre-mortem, reduced flexibility.         | Add friction, slow-check, reduce intensity, delay irreversible action.  |
| **Low activation + low vigilance**    | Under-mobilised loop state.                          | Poor initiation, low search energy, weak action.                       | Activate gently: movement, daylight, easy entry task, then reassess.    |

## 6. Revised state labels for the app / framework

I would now use two levels: **measured state** and **G-loop risk label**.

### Measured state layer

```text
Cognitive control:
  Regulated
  Slow compensatory
  Globally overloaded
  Fast brittle

Engagement-vigilance:
  Stable
  Low / unstable

Autonomic/body-budget:
  Good reserve
  Elevated load / low reserve
  Settling / recovering
  Rising mobilisation
  Low activation

Personal baseline:
  Typical
  Mild deviation
  Strong deviation
  Repeated drift
```

### G-loop risk layer

```text
Full-loop ready
Support-constrained
Inference-constrained
Closure-constrained
Action-too-fast
Scale-too-large
Vigilance-limited
Body-budget-limited
Banking-risk
Baseline-drift
```

This is cleaner than trying to turn every combined pattern into a separate “state”.

## 7. Practical decision rules

The rules should be:

```text
If regulated + stable vigilance + good reserve:
  run a meaningful G-loop; increase challenge if useful.

If regulated + autonomic strain:
  run a shorter loop; protect F★; bank, then recover.

If slow compensatory:
  preserve accuracy but reduce checking cost; use time-boxed closure.

If overloaded:
  reduce scale before action; simplify variables; scaffold Gf inference.

If fast brittle:
  do not trust speed as readiness; add disconfirmation before closure.

If low vigilance:
  reduce SR horizon; use activation or recovery before high-load inference.

If low activation + low vigilance:
  activate gently before demanding work.

If rising mobilisation + fast brittle:
  treat as brittle activation; slow-check and delay irreversible decisions.

If poor cognition + preserved autonomic reserve:
  use cognitive scaffolding rather than assuming physiological stress.

If repeated baseline drift:
  shift from single-loop routing to workload/recovery review.
```

## 8. Most compact formulation

```text
Zone Coach does not only classify readiness.

It estimates which part of the strategic G-loop is currently least reliable:

support estimation,
F★ protection,
MI target formation,
Gf model-building,
SR horizon selection,
right-sized closure,
external action,
prediction-error learning,
or deep Gc banking.
```

And the final state-framework slogan could be:

```text
MindState shows the control style.
Vigilance shows whether the loop can be sustained.
BodyState shows the F★ / support cost.
Baseline shows whether this is state noise or drift.

Together they predict where the G-loop may fail,
and what correction makes the next loop safer, clearer and more bankable.
```
