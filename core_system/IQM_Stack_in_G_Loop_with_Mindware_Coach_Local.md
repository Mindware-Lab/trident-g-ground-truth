# Mapping the IQM Stack to the Strategic G-Loop

## Core principle

The IQM stack is the **cognitive engine** of the G-loop.

The G-loop is the **control cycle** that decides:

```text
what state the learner is in
what must be protected
what larger arc matters
what objective is being trained
which cognitive layer is currently limiting the loop
what action should be taken
what feedback means
what should be banked as portable Gc
```

So the relationship is:

```text
G-loop = adaptive control architecture

IQM stack = measurable cognitive-capacity stack inside the loop
```

The IQM stack should not be presented as five unrelated scores. It should be presented as the layered machinery by which a person converts uncertainty into tested action and then banks what survives.

---

# 1. High-level mapping

| Strategic G-loop step                 | Practical function                                      | IQM stack expression                                                                                                  |
| ------------------------------------- | ------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **0. State and support estimate**     | Is the user in a trainable state?                       | Zone Coach / readiness check / session-quality flags / fatigue and timing checks                                      |
| **1. Protect F★**                     | Do not damage the defended base                         | Reset, rest, lower-load routing, avoid overtraining, claims and safety boundaries                                     |
| **2. Choose larger strategic arc**    | What future capability or option is being expanded?     | Exam resilience, AI workflow cognition, reasoning transfer, cognitive resilience, professional performance, longevity |
| **3. Define MI target / objective**   | What counts as success in this loop?                    | Current training focus, bottleneck target, graph family, relation family, wrapper-recovery target                     |
| **4. Gf model-building / inference**  | Build a model of the situation and infer possible paths | Reasoning layer, Path Prediction, relation-family recovery, graph-derived reasoning                                   |
| **5. Select scale and SR horizon**    | How far ahead should the loop simulate?                 | Path Prediction horizon, session type, benchmark cycle, weekly relation-family rotation                               |
| **6. Choose closure point**           | What should be completed or tested now?                 | Complete today’s block, run focused bottleneck session, do reasoning bridge, perform transfer check                   |
| **7. Pre-mortem / probability check** | What could break the loop?                              | Lure resistance, rare-vs-invalid checks, cannot-tell reasoning, timing quality, confidence labels                     |
| **8. Act externally / perturb niche** | Move beyond internal training into the environment      | Real-Life Transfer Check, micro-missions, implementation intentions, workflow prompts, coaching actions               |
| **9. Prediction-error evaluation**    | Did reality confirm or disconfirm the model?            | Scores, trends, transfer checks, Matrix Reasoning Benchmark, outcome logs, personal baseline update                   |
| **10. Deep Gc banking**               | What portable structure survives?                       | Delayed re-checks, Transfer Score, wrapper recovery, reusable mindware scripts, banked rules and routines             |

# 2. Mapping the five IQ Coach capacities to the G-loop

| IQM layer   | Public label          | G-loop role                                        | What it contributes                                                                                        |
| ----------- | --------------------- | -------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| **Layer 1** | **Attention Control** | Entry condition for Gf modelling                   | Extracts the relevant signal under time pressure. Without this, the loop cannot reliably select variables. |
| **Layer 2** | **Relational Memory** | Active relation maintenance                        | Holds graph-valid states, relations or edges across delay so Gf can compare, update and test them.         |
| **Layer 3** | **Binding Memory**    | Context and role preservation                      | Keeps relation-colour-context conjunctions stable so the loop does not confuse what belongs with what.     |
| **Layer 4** | **Path Prediction**   | SR horizon and future-state simulation             | Learns what usually follows, what is rare, what is invalid and whether a path can still reach the target.  |
| **Layer 5** | **Reasoning**         | Explicit Gf inference and global-workspace control | Recovers the same graph relation in symbolic, nonsense-semantic and domain-semantic reasoning.             |

Compactly:

```text
Attention Control extracts the signal.
Relational Memory holds the relation.
Binding Memory preserves the conjunction.
Path Prediction predicts the path.
Reasoning explains what follows.
```

This matches the IQ Coach system principle: the goal is not local game fluency, but whether the relation survives noise, delay, binding demands, wrapper change, future-state prediction and explicit reasoning.

# 3. The IQM stack as an inner G-loop

A single IQM training session can be understood as a **micro-G-loop**.

```text
State gate
→ Attention Control
→ Relational Memory
→ Binding Memory
→ Path Prediction
→ Reasoning
→ result interpretation
→ next action
→ delayed re-check
```

Mapped to the G-loop:

| Session phase             | G-loop function                          |
| ------------------------- | ---------------------------------------- |
| Readiness / quality check | Estimate state and support               |
| Attention Control         | Stabilise evidence extraction            |
| Relational Memory         | Maintain relation under delay            |
| Binding Memory            | Preserve context and conjunction         |
| Path Prediction           | Simulate likely / rare / blocked futures |
| Reasoning bridge          | Make the graph relation explicit         |
| Result screen             | Prediction-error evaluation              |
| Recommendation            | Right-sized closure point                |
| Transfer / delayed check  | Deep Gc banking                          |

So an 18-minute IQ Coach session is not merely:

```text
five mini-games
```

It is:

```text
one adaptive loop through signal extraction, relation maintenance, binding, prediction, reasoning, feedback and banking
```

---

# 4. Where the G-loop adds value beyond the stack

The IQM stack tells us **what capacity is being trained**.

The G-loop tells us **why, when and how that capacity should be deployed**.

| IQM stack alone                                    | IQM stack inside the G-loop                                  |
| -------------------------------------------------- | ------------------------------------------------------------ |
| Trains attention, memory, prediction and reasoning | Routes training by state, objective, scale and feedback      |
| Produces capacity scores                           | Interprets scores as loop constraints or bottlenecks         |
| Shows performance in app tasks                     | Tests whether structures survive wrapper change and real use |
| Tracks improvement                                 | Banks only what survives delay and transfer                  |
| Can become a training game                         | Becomes a cognitive workflow and transfer system             |

This is important strategically because the Trident-G protocol explicitly warns against rewarding trained-task gain alone. A structure should be banked only when it survives wrapper change, foils, mixed presentation, SR-style prediction, explicit reasoning or mindware selection, niche deployment where relevant, and delayed re-check.

# 5. Mapping bottlenecks to G-loop failure points

| G-loop failure                                 | Likely IQM bottleneck        | App signal                                                             |
| ---------------------------------------------- | ---------------------------- | ---------------------------------------------------------------------- |
| Cannot stabilise task-relevant evidence        | Attention Control bottleneck | Low Attention Control; unstable timing or extraction                   |
| Cannot hold relation across delay              | Relational Memory bottleneck | Wrong-lag lure errors; low relation-bit steps                          |
| Cannot preserve role/context bindings          | Binding Memory bottleneck    | Partial-match errors; same relation wrong colour/context               |
| Cannot simulate future path                    | Path Prediction bottleneck   | Good one-step prediction but poor two-step reachability                |
| Cannot make relation explicit                  | Reasoning bottleneck         | Good visual graph learning but poor symbolic/nonsense/domain reasoning |
| Local scores improve but portability fails     | Transfer bottleneck          | Low wrapper recovery, delayed recovery or Real-Life Transfer Check     |
| Good same-session performance disappears later | Consolidation bottleneck     | Poor delayed re-check; unstable Transfer Score                         |

This should be the bridge between the score dashboard and the G-loop coaching interpretation.

# 6. Product architecture implication

The clean product hierarchy becomes:

```text
G-loop control layer
├── State and readiness routing
├── F★ protection / safe-load rules
├── Strategic arc and objective selection
├── IQM capacity stack
│   ├── Attention Control
│   ├── Relational Memory
│   ├── Binding Memory
│   ├── Path Prediction
│   └── Reasoning
├── Transfer and proof layer
│   ├── Wrapper Recovery
│   ├── Delayed Recovery
│   ├── Real-Life Transfer Check
│   ├── Transfer Score
│   └── Matrix Reasoning Benchmark
└── Deep Gc banking
    ├── reusable rules
    ├── mindware scripts
    ├── personal baselines
    ├── delayed re-entry checks
    └── real-world deployment records
```

The IQ Coach app spec already separates the five core capacities from transfer and proof measures such as Real-Life Transfer Check, Transfer Score and Matrix Reasoning Benchmark.  The G-loop explains why this separation matters: core capacities estimate structured information handling, while proof measures test whether that structure actually becomes portable.

# 7. Mindware Coach Local v0: schema-controlled mission generator

## 7.1 Product role

**Mindware Coach Local v0** is a downloadable local AI layer that sits **above** the IQM capacity stack and **below** open-ended coaching or human consultation.

It should not be a free-chat bot and should not be treated as a psychometric, clinical or scoring system.

Its first job is narrow:

```text
read state / profile / context
→ select the relevant mindware meta-function
→ generate a short mission and H-AGI prompt
→ remain inside IQ Mindware claims boundaries
```

In the G-loop, Mindware Coach Local v0 belongs mainly at:

```text
Gf model-building / inference
→ pre-mortem / probability check
→ external action / niche perturbation
→ prediction-error reflection
→ deep Gc banking
```

It turns app-native signals into **small, usable cognitive moves** without changing the underlying score, diagnosis, norming or bottleneck engines.

---

## 7.2 Architecture principle

Use a strict separation:

```text
IQM deterministic engine
= graph generation, scoring, capacity estimates, bottleneck rules, confidence labels, claims boundaries

Mindware Coach Local model
= coaching language, mission generation, H-AGI prompts, reflection prompts and domain wrapping
```

The local model may interpret a profile, but it must not create or alter canonical estimates.

It may say:

```text
Your current profile suggests a useful mission would be to test whether a conclusion really follows.
```

It must not say:

```text
Your reasoning ability is low.
Your IQ has changed.
You have an attention problem.
This proves transfer.
```

The rule is:

```text
The model explains and personalises.
The measurement engine scores.
```

---

## 7.3 Required input object

The model should receive a structured state/profile/context object rather than raw unrestricted chat.

Example:

```json
{
  "user_state": {
    "readiness_label": "Narrow Focus",
    "session_quality": "good",
    "confidence": "moderate"
  },
  "capacity_profile": {
    "attention_control": "standard",
    "relational_memory": "moderate",
    "binding_memory": "calibrating",
    "path_prediction": "low_confidence",
    "reasoning": "cannot_tell_lures_elevated"
  },
  "current_context": {
    "domain": "AI workflow cognition",
    "task": "reviewing an AI-generated report",
    "stakes": "medium",
    "time_pressure": "moderate"
  },
  "available_actions": [
    "train",
    "reset",
    "widen",
    "slow_check",
    "reasoning_mission",
    "implementation_intention"
  ]
}
```

This protects the architecture because the local AI is not guessing the whole situation from free text. It is responding to a structured control-state produced by the IQM system.

---

## 7.4 Required output schema

Mindware Coach Local v0 should emit a constrained JSON object.

Example:

```json
{
  "mode": "mission",
  "meta_function": "argumentation",
  "relation_family": "context_constraint",
  "user_facing_title": "Find the cannot-tell point",
  "coach_message": "Today’s useful move is to slow down one conclusion and check whether it really follows from the evidence given.",
  "mission_steps": [
    "Choose one claim from the AI output.",
    "List the evidence directly provided.",
    "Mark what follows, what does not follow and what cannot be known yet.",
    "Rewrite the claim with the uncertainty made explicit."
  ],
  "h_agi_prompt": "Which parts of this conclusion are directly supported by the evidence, and which parts are only plausible extensions?",
  "claims_boundary": "This is a reasoning support exercise, not a diagnosis, IQ estimate or proof of transfer.",
  "gc_banking_prompt": "What rule or warning sign should you reuse next time?"
}
```

The user can see a friendly version of this output, but the app should store the structured object for audit, versioning and later analysis.

## 7.5 Meta-functions supported in v0

The first local model should support a small set of mindware meta-functions.

| Meta-function | G-loop role | Example mission |
| ------------- | ----------- | --------------- |
| **Comprehension** | Build the current state model | Identify the claim, variables, context and missing information. |
| **Argumentation** | Test whether conclusions follow | Separate evidence, assumption, conclusion and cannot-tell points. |
| **Decision-making** | Choose a closure point | Compare options, constraints, risks and smallest safe tests. |
| **AI-output supervision** | Protect source/context binding | Check whether AI claims remain tied to evidence and context. |
| **Strategic action** | Perturb the niche | Convert a reasoning result into a small external action. |
| **Implementation intention** | Increase cue-policy coupling | Create an if-then plan for using the mindware move later. |
| **Reflection / Gc banking** | Stabilise what survived | Name the reusable rule, lure or boundary condition. |

This keeps v0 aligned with the G-loop without trying to become a general-purpose coach.

---

## 7.6 Mapping to the IQM stack

Mindware Coach Local v0 should not be presented as a sixth core capacity.

It is a **deployment and banking layer** that translates the five capacities into practical missions.

| IQM stack signal | Local coach interpretation | Mission direction |
| ---------------- | -------------------------- | ----------------- |
| Low or unstable Attention Control | The loop may not have a stable signal base. | reset, simplify, reduce load, extract the key cue |
| Relational Memory bottleneck | Relations may not be surviving delay or lures. | hold one relation, compare later, check wrong-lag lures |
| Binding Memory bottleneck | Features or claims may be separating from context. | bind claim-source-context, check partial matches |
| Path Prediction bottleneck | The next-state map may be shallow. | ask what usually follows, what is rare, what is blocked |
| Reasoning bottleneck | The graph may not be transferring into explicit inference. | valid / invalid / cannot tell, same relation, best conclusion |
| Transfer bottleneck | Local performance may not yet be portable. | use a mission in a changed context and re-check later |

The local coach therefore helps answer:

```text
What mindware move should the user practise now,
given this state, profile and context?
```

---

## 7.7 Claims and safety boundaries

Mindware Coach Local v0 must inherit the IQM claims boundary.

It may say:

```text
This mission practises checking whether a conclusion follows.
This prompt helps you test assumptions with an AI assistant.
This exercise may help you make the trained relation more usable in a real task.
```

It must not say:

```text
This raises your IQ.
This proves your intelligence has improved.
This diagnoses attention, anxiety, ADHD, depression or cognitive impairment.
This is therapy.
This should be used for hiring, promotion, admission, exclusion or ranking.
```

For organisational deployments, the local coach should remain individual-facing unless the user explicitly chooses to share a mission summary. It should not transmit raw scores, private reflections or individual coaching outputs to a manager dashboard.



## 7.9 Build priority for v0

Build v0 in this order:

```text
1. Define input and output schemas.
2. Build mission-template library.
3. Build claims-boundary and refusal library.
4. Create prompt-only prototype using a small local instruction model.
5. Add local retrieval over approved IQ Mindware docs and mission templates.
6. Create gold examples for each meta-function and domain wrapper.
7. Fine-tune only after the schema and evaluation suite are stable.
8. Quantise and package for local download.
9. Integrate as an optional layer inside Mindware Flow / IQ Coach.
```

The first release should do four things well:

```text
1. Read a state/profile/context object.
2. Select the right mindware meta-function.
3. Generate a short mission and H-AGI prompt.
4. Stay inside IQ Mindware claims boundaries.
```

Everything else belongs after v0.

## 7.10 G-loop summary

Mindware Coach Local v0 is the bridge from measured capacity to applied cognitive action.

```text
IQM stack measures the control chain.
Transfer checks test whether it survives.
Mindware Coach Local suggests the next portable move.
Delayed reflection helps bank what worked as reusable Gc.
```

In compact form:

```text
score → bottleneck → meta-function → mission → H-AGI prompt → action → reflection → Gc banking
```

The strategic value is that IQ Mindware becomes not only a training and proof system, but a **local cognitive mission system**: a way to turn cognitive profile data into private, claims-safe, day-to-day mindware practice.

# 8. Recommended spec wording

Add a section like this to the IQM / IQ Coach specs:

## G-Loop Mapping

IQ Coach implements the cognitive-capacity layer of the strategic G-loop.

The strategic G-loop is the higher-order adaptive cycle:

```text
state → constraint → arc → objective → Gf inference → scale → closure → action → feedback → deep Gc
```

IQ Coach supplies the measurable cognitive operations inside that cycle:

```text
Attention Control
→ Relational Memory
→ Binding Memory
→ Path Prediction
→ Reasoning
```

These layers allow the system to estimate whether the user can:

```text
extract the signal
hold the relation
bind relation to context
predict future states
reason explicitly with the same structure
recover the relation after wrapper change and delay
```

The app should therefore not present the five capacities as isolated brain-training scores. It should present them as a vertical control chain supporting adaptive G-loop completion.

The central product question is:

```text
Can the user convert uncertainty into a tested relation,
use that relation across cognitive layers,
act or transfer appropriately,
and bank what survives as portable future capability?
```

A session is complete only when it has generated at least one of the following:

```text
a clearer capacity estimate
a better bottleneck hypothesis
a recovered relation
a successful wrapper transition
a resisted lure
a stronger path prediction
a reasoning transfer signal
a delayed recovery signal
a real-life application signal
a reusable mindware rule
```

The app should reward the recoverable control chain, not merely correct responses.

The one refinement I would make is to keep **Zone Coach / Mindware Flow** partly outside the five-capacity IQ Coach stack. It maps mainly to **state/support estimation** and **F★ protection**, whereas IQ Coach proper maps to the inner cognitive engine of the loop. That keeps the architecture clean:

```text
Zone Coach = Can I run the loop safely now?
IQ Coach = Which cognitive layer is limiting the loop?
Reasoning / Transfer / Missions = Can the loop act, update and bank?
```
