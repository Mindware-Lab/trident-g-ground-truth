# Attention Coach Horizontal Transfer Protocol

## Components B and C: Evidence Control and Decision Timing

**Version:** v0.2
**Status:** Draft implementation protocol
**Scope:** Two additional Attention Coach component tasks aligned with the updated carrier × reference-computation architecture
**Claims boundary:** Behavioural training and measurement protocol; non-clinical, non-diagnostic, and not evidence by itself of broad real-world far transfer

---

## 1. Three-component Attention Control architecture

Attention Coach contains three related but dissociable control tasks:

| Component             | Public label         | Plain-language question                                        |
| --------------------- | -------------------- | -------------------------------------------------------------- |
| Existing signal task  | **Signal Control**   | Can you pick out the relevant relation under noise?            |
| New accumulation task | **Evidence Control** | Can you combine evidence and update when reliability changes?  |
| New pressure task     | **Decision Timing**  | Can you choose the right moment to commit under time pressure? |

The overarching control sequence is:

```text
SIGNAL
Select the relevant information.

ACCUMULATE
Integrate enough evidence and update the current model.

COMMIT
Stop sampling and act at the appropriate threshold.
```

Or:

> **Select the right information, accumulate enough of it, and commit at the right time.**

This follows the July 2026 overview, which defines the repeated control problem as extraction, evidence accumulation and commitment, and treats these as related but separable functions.  The overview also distinguishes representational capacity, evidence updating and stopping control as separate sources of failure.

The three internal component estimates are:

```text
C_attention
Signal extraction and representational capacity

A_attention
Evidence accumulation, updating and reliability adaptation

T_attention
Threshold adjustment and decision timing
```

The public dashboard should continue to show one top-level capacity:

```text
Attention Control
```

The three tasks are subscores and training focuses, not three new top-level cognitive layers.

# 2. Updated horizontal-transfer architecture

## 2.1 The two perturbation dimensions

The four atomic conditions are generated by crossing two dimensions.

### Carrier

```text
arrow
optic flow / local motion
```

### Reference computation

```text
fixed axis
common centre
```

This gives:

| Atomic condition | Carrier      | Reference computation | Relation     |
| ---------------- | ------------ | --------------------- | ------------ |
| `arrow_abs`      | arrows       | fixed screen axis     | LEFT / RIGHT |
| `flow_abs`       | local motion | fixed screen axis     | LEFT / RIGHT |
| `arrow_rel`      | arrows       | common centre         | IN / OUT     |
| `flow_rel`       | local motion | common centre         | IN / OUT     |

The portable invariant is not one identical response label across all four conditions. It is:

> **Recover the majority relation under the currently active reference computation, then regulate evidence gathering and commitment appropriately.**

More precisely:

```text
Carrier transfer:
same LEFT / RIGHT relation
different visual carrier

Reference-computation transfer:
same majority-control operation
different coordinate rule

Held-out composition:
new carrier
×
previously learned common-centre computation
```

# Part I — Evidence Control

## 3. Construct

### 3.1 Technical definition

Evidence Control estimates the user’s ability to:

```text
combine sequential sensory evidence
update confidence
infer whether the evidence stream is currently reliable
adapt after an unannounced reliability change
avoid premature commitment
avoid unnecessary continued sampling
```

It asks:

> **Can the user combine several imperfect observations, infer how trustworthy the stream is, and decide when enough evidence has accumulated?**

The July 2026 overview describes this attention-level task as sequential evidence arriving with changing reliability, with measures including evidence sensitivity, drift or accumulation rate, temporal weighting, change-point adaptation and confidence calibration.

### 3.2 Internal label

```text
component: accumulate
model family: AEC
technical construct: Adaptive Evidence Control
primary estimate: A_attention
```

### 3.3 What it is not

Evidence Control is not:

```text
another single-display bits/sec score
a working-memory span task
a disorder score
a direct measure of real-world judgement
```

Signal Control remains the lower-level extraction anchor.

---

## 4. Core task

Each episode contains:

```text
one hidden target relation
+
one hidden reliability context
+
a sequence of brief masked majority displays
+
a user-controlled stopping decision
```

Example:

```text
Hidden target:
LEFT

Hidden reliability:
stable

Sample 1:
4 LEFT / 1 RIGHT

Sample 2:
4 RIGHT / 1 LEFT

Sample 3:
4 LEFT / 1 RIGHT

Sample 4:
4 LEFT / 1 RIGHT
```

The user may select LEFT or RIGHT after any sample.

If the user has not responded by the maximum sample count, the task requests a forced choice.

### 4.1 User instruction

> **Watch the evidence. Choose when you have seen enough.**

Tutorial addition:

> Some evidence streams are less reliable than others. You will not always be told when this changes.

---

## 5. Evidence contexts

Use at least two latent contexts.

| Context    | Meaning                                              | Illustrative starting reliability |
| ---------- | ---------------------------------------------------- | --------------------------------: |
| **Stable** | Individual samples usually support the hidden target |                               .85 |
| **Noisy**  | Misleading samples occur more frequently             |                               .65 |

These probabilities are initial calibration values, not permanent canonical parameters.

The context should last for a variable run of episodes and then change without an explicit announcement.

Both transitions must be represented:

```text
stable → noisy
noisy → stable
```

Carrier and reference computation must not predict reliability.

Unsafe design:

```text
arrows usually stable
optic flow usually noisy
```

Required design:

```text
P(reliability context | wrapper)
is balanced by construction
```

---

## 6. Episode structure

```text
fixation
→ evidence sample
→ mask
→ response opportunity
→ if no response, next evidence sample
→ response or forced choice
→ optional confidence probe
→ feedback
→ ITI
```

Recommended configurable fields:

```text
maximum samples
sample interval
single-sample exposure
mask duration
confidence-probe rate
context duration
context hazard rate
```

The perceptual sample should initially be easier than the user’s Signal Control threshold. This prevents perceptual extraction failure from being mistaken for accumulation failure.

---

## 7. Evidence Control metrics

### 7.1 Required metrics

| Metric                     | Operational meaning                                                    |
| -------------------------- | ---------------------------------------------------------------------- |
| **Evidence Accuracy**      | Whether the hidden target is correctly recovered                       |
| **Samples to Decision**    | Number of samples observed before commitment                           |
| **Evidence Sensitivity**   | Degree to which cumulative evidence changes choice probability         |
| **Reliability Adaptation** | Difference in stopping behaviour between stable and noisy contexts     |
| **First-Trial Adaptation** | Behaviour on the first episode following an unannounced context change |
| **Context-Switch Lag**     | Episodes required for behaviour to reflect the new context             |
| **Temporal Weighting**     | Relative influence of early and recent evidence                        |
| **Context Hysteresis**     | Persistence of the previous context policy                             |
| **Premature Commitment**   | Commitment before adequate evidence has accumulated                    |
| **Excess Sampling**        | Continuing to sample after evidence is sufficiently stable             |
| **Decision Variability**   | Variation in evidence requirements under equivalent conditions         |
| **Confidence Calibration** | Correspondence between confidence and correctness                      |

### 7.2 Utility

A configurable utility score may be estimated:

```text
Evidence Utility
=
decision outcome
-
sampling cost
```

The utility model must not reward indefinite waiting merely because it improves accuracy.

### 7.3 Fixed-policy exclusion

The following strategies cannot satisfy the portability gate:

```text
always respond after one sample
always wait for all samples
always use one threshold regardless of reliability
```

A successful Evidence Control policy must adapt to the inferred reliability context.

# 8. Evidence Control horizontal-transfer cycle

## Stage 1 — Base

```text
wrapper:
arrow_abs

reference computation:
fixed axis

relations:
LEFT / RIGHT
```

Train until there is stable evidence for:

```text
adequate final accuracy
evidence-sensitive choice
different thresholds for stable and noisy contexts
controlled premature commitment
controlled excess sampling
acceptable context-switch lag

```

### Optional diagnostic carrier probes

Before formal transfer:

```text
5–10% flow_abs episodes
```

These may estimate readiness but cannot:

```text
count as recovery

bank transfer
```

---

## Stage 2 — Carrier perturbation

Introduce:

```text
flow_abs
```

The target remains:

```text
LEFT / RIGHT
```

The relation and evidence-control policy remain constant; only the carrier changes.

A formal transition block may begin with:

```text
80% arrow_abs
20% flow_abs
```

During this perturbation, freeze:

```text
single-sample perceptual demand
reliability probabilities
context duration
maximum sample count
sampling cost
feedback schedule
```

---

## Stage 3 — Measure the dip

Measure separate first-contact effects for:

```text
final decision accuracy
samples to decision
evidence sensitivity
reliability adaptation
context-switch lag
evidence utility
```

A user may maintain accuracy while temporarily losing reliability-sensitive threshold control.

Example:

```text
accuracy remains high
but stable and noisy contexts now produce the same sampling threshold
```

That is a policy-transfer dip even if the final answer remains correct.

The first few `flow_abs` episodes are descriptive. Controller advancement requires a larger matched target-wrapper sample.

---

## Stage 4 — Carrier recovery

Train blocked `flow_abs` episodes until:

```text
evidence accuracy recovers
reliability adaptation reappears
premature commitment is controlled
excess sampling is controlled
context-switch lag stabilises
timing quality is acceptable
```

Do not make unrelated difficulty harder during recovery.

---

## Stage 5 — Return to base

Return to:

```text
arrow_abs
```

Test whether learning `flow_abs` disrupted the original evidence-control policy.

Compute:

```text
return strength
=
post-perturbation arrow_abs performance
/
pre-perturbation arrow_abs performance
```

Return evidence must distinguish:

```text
sufficient and passes
sufficient and fails
insufficient — collect more evidence
```

---

## Stage 6 — Carrier mixing

Progress through:

```text
80/20
→ 60/40
→ 50/50
→ random arrow_abs / flow_abs
```

Reliability context must vary independently of carrier.

The user must infer stream reliability from the evidence statistics rather than from the wrapper.

Mixed stability should be evaluated relative to blocked-wrapper performance.

---

## Stage 7 — Reference-computation perturbation

Introduce:

```text
arrow_rel
```

The task now uses:

```text
IN / OUT
```

The invariant is the evidence-control operation:

```text
extract the active majority relation
→ integrate sequential evidence
→ adjust the stopping threshold to reliability
```

Use an explicit pre-stimulus rule cue:

```text
LEFT / RIGHT
```

or:

```text
IN / OUT
```

The cue must occur before the timed stimulus sequence and must not enter the perceptual exposure calculation.

---

## Stage 8 — Reference-computation mixing

Mix:

```text
arrow_abs + arrow_rel
```

with explicit rule cues.

The task must balance:

```text
fixed-axis and common-centre trials
stable and noisy contexts
target relation
response side
```

The user should not be able to infer reliability from the current rule cue.

---

## Stage 9 — Held-out composition

The protected composition is:

```text
flow_rel
=
motion carrier
+
common-centre IN / OUT computation
```

Before the formal held-out event:

```text
no flow_rel tutorial
no flow_rel free play
no flow_rel example animation
no mixed block containing flow_rel
```

The initial `flow_rel` episodes occur without feedback-enabled recovery contaminating the held-out estimate.

Measure:

```text
compositional accuracy
sampling-policy transfer
reliability adaptation
context-switch lag
premature commitment
excess sampling
```

---

## Stage 10 — Full-factorial Evidence Control

Randomly mix:

```text
arrow_abs
flow_abs
arrow_rel
flow_rel
```

under both:

```text
stable
noisy
```

This creates eight core cells:

```text
2 carriers
× 2 reference computations
× 2 reliability contexts
```

The benchmark should include cases where:

```text
wrapper changes but reliability stays constant
reliability changes but wrapper stays constant
both change
neither changes
```

This tests whether the user can distinguish surface change from statistical-context change.

---

## Stage 11 — Delayed Evidence Control re-check

Use fresh delayed all-four mixed evidence only.

The delayed block must provide adequate new coverage of:

```text
all four wrappers
both reliability contexts
both context-switch directions
```

Completing a delayed session is not itself evidence that Evidence Control is portable.

---

## Stage 12 — Evidence Control banking

Bank `A_attention` portability only when:

```text
blocked-wrapper recovery passes
return-to-base passes
mixed-wrapper stability passes
held-out composition passes
full-factorial reliability adaptation passes
fresh delayed mixed evidence passes
timing quality is acceptable
```

# Part II — Decision Timing

## 9. Construct

### 9.1 Technical definition

Decision Timing estimates the user’s ability to:

```text
adjust stopping thresholds to available time
balance evidence value against delay cost
slow down selectively rather than globally
anticipate the time needed to complete a response
avoid premature closure
avoid missed deadlines
```

It asks:

> **Can the user choose the right moment to stop sampling and commit under time pressure?**

The overview characterises the attention-level pressure task as varying response deadlines and measuring speed–accuracy adjustment, response threshold, urgency, deadline misses, conflict errors and selective slowing.

### 9.2 Internal label

```text
component: commit
model family: DTC
technical construct: Decision Timing Control
primary estimate: T_attention
```

### 9.3 Relation to Evidence Control

Evidence Control asks:

```text
How much evidence is enough under changing reliability?
```

Decision Timing asks:

```text
How much evidence is enough given the available time,
cost of delay and time required to act?
```

---

# 10. Decision Timing task modes

## 10.1 Mode A — Timed Commit

This is the recommended MVP.

The user sees sequential evidence samples and a visible deadline.

They may choose the active relation at any point.

Possible outcomes:

```text
correct
incorrect
omitted / deadline missed
```

User instruction:

> **Use the evidence, but do not miss the deadline.**

This mode measures adaptive stopping under time pressure without requiring a separate response-policy transition.

---

## 10.2 Mode B — React-to-Fallback Switch

This is a later benchmark mode.

The user begins in evidence-waiting mode.

They may press:

```text
DECIDE NOW
```

The system then:

```text
stops presenting evidence
→ applies a short variable action-gate delay
→ activates the answer options
→ requires the final response before the deadline
```

This creates an explicit transition from:

```text
wait for more evidence
→ implement a fallback commitment
```

User instruction:

> **Wait while the evidence is useful. Switch early enough to complete your decision.**

This mode is particularly useful for measuring late switching and threshold hysteresis.

---


# 12. Decision Timing episode structure

## Timed Commit

```text
fixation
→ deadline cue
→ sequential evidence begins
→ user commits at chosen point
→ correct / incorrect / omission
→ feedback
```

## React-to-Fallback Switch

```text
fixation
→ deadline cue
→ evidence-waiting state
→ user initiates DECIDE NOW
→ switch/action delay
→ answer options activate
→ final choice or omission
→ feedback
```

The deadline must use a monotonic clock.

# 13. Decision Timing metrics

## 13.1 Required metrics

| Metric                     | Operational meaning                                               |
| -------------------------- | ----------------------------------------------------------------- |
| **Deadline Accuracy**      | Correct completed decisions by deadline regime                    |
| **Omission Rate**          | Failure to complete a response before the deadline                |
| **Decision Time**          | Time from episode onset to commitment                             |
| **Evidence at Commitment** | Accumulated evidence when commitment begins                       |
| **Threshold Adjustment**   | Difference in evidence threshold across deadline regimes          |
| **Selective Slowing**      | Additional time used for ambiguous rather than easy episodes      |
| **Urgency Slope**          | Increase in response probability as the deadline approaches       |
| **Fast Conflict Errors**   | Early errors when evidence is internally conflicting              |
| **Late-Closure Cost**      | Utility lost through unnecessary delay or omission                |
| **Threshold Variability**  | Variation in stopping threshold under matched conditions          |
| **Threshold Hysteresis**   | Persistence of the previous deadline policy after a regime change |

## 13.2 Mode B switch metrics

| Metric                         | Meaning                                                   |
| ------------------------------ | --------------------------------------------------------- |
| **Fallback Switch Point**      | Time remaining when waiting is abandoned                  |
| **Switch Margin**              | Deadline margin when the fallback begins                  |
| **Effective Switch Latency**   | Time from switch initiation to completed answer           |
| **Switch-Latency Variability** | Variability in implementation time                        |
| **Late-Switch Cost**           | Omissions caused by switching too late                    |
| **Premature-Fallback Cost**    | Errors caused by abandoning useful evidence too early     |
| **Switch-Cost Adaptation**     | Adjustment when switching becomes slower or more variable |

## 13.3 Utility

```text
Decision Utility
=
correct reward
-
incorrect cost
-
delay cost
-
omission cost
```

Indiscriminate speed must not produce a high score.

Indiscriminate waiting must not produce a high score.

# Part III — Shared implementation rules

## 15. One-dimensional perturbation

Wrapper difficulty, reliability difficulty and temporal difficulty must be manipulated separately.

### During wrapper perturbations, freeze:

```text
perceptual strength
majority ratio
reliability schedule
context volatility
maximum samples
sample cost
deadline
deadline variability
switch cost
feedback schedule
```

### During reliability perturbations, freeze:

```text
wrapper
reference computation
deadline
perceptual difficulty
```

### During deadline perturbations, freeze:

```text
wrapper
reference computation
reliability schedule
perceptual difficulty
```

---

## 16. Rule cues

When fixed-axis and common-centre conditions are mixed, present an explicit pre-stimulus cue:

```text
LEFT / RIGHT
```

or:

```text
IN / OUT
```

The cue:

```text
must be visually unambiguous
must appear before timed evidence begins
must not count towards exposure duration
must not predict reliability or deadline
```

Carrier-only changes do not require a different relation-rule cue.

---

## 17. Shared held-out exposure

The strongest protocol uses one global exposure ledger across Signal Control, Evidence Control and Decision Timing.

A visual condition shown in one component is no longer perceptually held out in another component.

Recommended design:

```text
first flow_rel exposure:
Signal Control held-out probe
→ Evidence Control held-out probe
→ Decision Timing held-out probe
```

All three occur before `flow_rel` feedback-enabled recovery begins.

This creates a matched **held-out triad benchmark**.

When this is not possible, distinguish:

```text
clean held-out visual composition
from
first use of an untrained component policy
```

Do not call the second one fully clean compositional transfer.

---

## 18. Separate component states

Maintain separate transfer states:

```text
signal transfer state
accumulation transfer state
commitment transfer state
```

Signal recovery cannot be used as evidence that Evidence Control or Decision Timing recovered.

Recommended component outcomes:

```text
portable
maintenance required
repair signal
repair accumulation
repair commitment
insufficient evidence
```

Do not average a failed component into an apparently successful general score.

---

## 19. Full Attention Control portability

The complete Attention Control policy is:

```text
recover the relevant relation
→ accumulate diagnostic evidence
→ calibrate confidence
→ stop at the appropriate threshold
→ act
→ reopen when prediction error warrants it
```

The overview treats this as a vertically reusable policy that can recur as the represented object changes from sensory signal to relation, binding, path and explicit conclusion.

Attention Control is portable only when:

```text
Signal Control survives wrapper change and delay.

Evidence Control survives wrapper and reliability-context change.

Decision Timing survives wrapper and deadline change.

The held-out composition survives.

Fresh delayed all-four mixed evidence passes.
```

# 20. Zone-pattern interpretation

The joint pattern across the three components supports more precise training profiles:

| Pattern                 | Signal Control                 | Evidence Control                   | Decision Timing                              |
| ----------------------- | ------------------------------ | ---------------------------------- | -------------------------------------------- |
| **Regulated**           | Good discrimination            | Evidence-sensitive and calibrated  | Threshold adapts to time and risk            |
| **Slow compensatory**   | Often preserved                | Excessive accumulation or checking | Threshold too high; late closure or omission |
| **Globally overloaded** | Reduced or unstable            | Fragmented updating                | Threshold erratic or unreachable             |
| **Fast brittle**        | May appear good on easy trials | Insufficient evidence use          | Threshold too low; premature response        |

This classification should be based on repeated joint evidence rather than mean response time and accuracy alone.

The key diagnostic distinction is:

```text
Low capacity:
the relation cannot be recovered reliably.

Accumulation failure:
the relation is available,
but evidence is not integrated appropriately.

Threshold failure:
the relation and evidence are adequate,
but action occurs too early or too late.
```

# 22. Claims boundary

Allowed:

> Attention Coach trains and measures signal extraction, sequential evidence use and decision timing under controlled conditions.

> It tests whether these operations recover across changes in carrier, reference computation, reliability context, deadline and delay.

Not allowed:

```text
This diagnoses ADHD, anxiety, pain or another condition.

This treats indecision.

This proves improved real-world judgement.

This proves IQ improvement.

This directly measures neural criticality.

This guarantees far transfer.
```
